3Notch V4 — Continuation Checkpoints

Opt-in, local continuation packets for recovering from rate limits, compaction, and agent switches without reading transcripts or blocking workflows.

Decision gate

● D1 Hybrid authoring: agent-drafted semantic checkpoints plus deterministic script fallbacks.
● D2 Hooks are non-blocking and consume only documented event fields plus Git metadata.
● D3 Default Claude events are SessionStart, TaskCreated, TaskCompleted, PostCompact, and rate-limit StopFailure; Stop is warned and opt-in.
● D4 Continuations remain immutable packets linked with tags and supersedes.
● D5 Remote MCP, transcript scraping, model routing, and automatic Git/network actions remain out of scope.

1. Policy and hook wiring

Add continuation configuration, explicit onboarding flags, safe local Claude settings merging, warnings, and doctor drift checks.

Accept: enabling, syncing, and disabling hooks preserves unrelated settings and missing configuration stays off.

2. Script recovery layer

Accumulate structured task events under ignored derived state and flush one packet on compaction, rate-limit failure, or opted-in Stop.

Accept: no transcript access, no workflow blocking, no per-task packet spam, and secret-scan failures remain non-fatal to Claude.

3. Packet and agent integration

Add tag/supersedes parity, stream-aware continuation chains, Git provenance, resume offers, and Claude agent drafting instructions.

Accept: agent drafts supersede fallbacks within one stream and resume never loads content without confirmation.

4. Verification and delivery

Complete unit, CLI, MCP, schema, e2e, and built-CLI checks; update public documentation; release as a focused PR.

Accept: all repository done criteria pass and the source branch contains only V4 work.

Deviations

The recovery layer stays useful precisely because it records less than a transcript and more than a Git diff.

Build debrief · 2026-07-19

What V4 actually changed

The approved D1–D5 plan shipped in four workstreams: policy and hook wiring, deterministic recovery, packet and agent integration, and verified delivery. The implementation was rebased over PRs #1–#3; the public website is delivered separately in PR #5.

User-visible behavior

Configure once, recover locally

notch onboard can wire Claude Code checkpoints in script, prompt, auto, or off mode. Setup preserves unrelated settings, warns before project-visible or high-frequency behavior, and notch doctor reports drift.

User-visible behavior

One fallback, not task spam

Task events accumulate locally. Post-compaction and rate-limit events can write one unreviewed fallback with structured progress and Git provenance. Optional Stop writes only when the deterministic state changed.

User-visible behavior

Resume only after confirmation

A later session offers the newest matching checkpoint once but does not load its body until the user approves. Agent-authored semantic checkpoints can supersede a mechanical fallback in the same stream.

Mechanism

The packet model stayed intact

Continuations remain ordinary immutable packets. Tags identify the stream and source event; supersedes orders checkpoints; project/private sensitivity controls the outbox; CLI and MCP expose matching tag and lineage fields.

Deviation

Recorded verbatim: “2026-07-19: The plan used colon-delimited tag examples. Existing packet metadata canonicalizes tags to slugs, so implementation uses stream-<slug> and source-<event> while preserving the same filtering semantics.”

Plan said: tags such as stream:<slug>. Territory forced: the shipped tag schema accepts canonical slugs, not colons. Lesson: extend an established serialization contract through its existing canonical form instead of creating a one-feature exception.

Intuition

The deterministic layer is a seat belt, not memory: it records only documented hook fields, accumulated tasks, and Git state. Semantic checkpoints carry the richer objective and decisions when an agent still has them. Both stay visible, local, unreviewed, and confirmation-gated because recovery quality should not erase the privacy boundary.

What did not change

  • No transcript reading, hidden project scraping, complete-conversation claim, hosted relay, remote MCP mode, model router, or cloud service.
  • No automatic commit, push, import, artifact copy, or packet transport.
  • No packet per Claude checklist item and no new continuation record type.
  • A fallback cannot recover reasoning absent from tasks, a compact summary, Git state, a final message, or an earlier semantic checkpoint.
  • The website is on a preview deployment and separate PR; production is unchanged until that PR is reviewed and merged after PR #4.
