01 · WHY

The repo survives. The working state often doesn't.

Your code, plans, wiki, and commits already hold durable project knowledge. The fragile part is what was in motion when a session compacted, hit a rate limit, or changed agents.

3Notch captures that missing continuation layer: the current objective, completed work, decisions, constraints, verification, blockers, relevant files, and exact next action.

Claude Code hooks can write a deterministic fallback from documented lifecycle fields and Git metadata. For a deliberate switch, the agent can prepare a richer semantic checkpoint. Both remain ordinary, local, reviewable packets.

03 · USE CASES

One packet model. Three useful boundaries.

3Notch does not replace your repo's durable knowledge. It records the state needed to continue, then adds files only when the work actually has to move.

01 · Recover

A session stops unexpectedly.

After compaction or a rate limit, a script fallback records documented task state and Git metadata without reading the transcript.

  PostCompact · rate_limit
  tasks + Git state
  ✓ unreviewed fallback
02 · Continue

You switch agents on purpose.

At a meaningful milestone, the agent drafts a semantic checkpoint with decisions, verification, blockers, and the exact next action.

  objective + decisions
  blockers + next action
  ✓ prompt or auto
03 · Hand off

The work crosses a boundary.

Add selected files and a destination when another tool, repo, machine, teammate, or browser chat needs a portable handoff.

Packet
  brief + selected files
  destination + next steps
  ✓ your transport
+

Resume is confirmation-gated. A new session offers the latest matching checkpoint once and loads it only after you approve. Nothing commits, pushes, imports, or travels automatically.

04 · PACKETS

Plain files, explicit context, no hidden memory.

Continuation checkpoints and portable handoffs use the same immutable packet model. Markdown remains the source of truth; selected artifacts are optional.

Local trail · Portable packet

Readable now. Useful to the next agent.

A checkpoint can be one Markdown record in .notch/. A cross-boundary handoff can add a manifest and selected artifacts, then pack into one hash-verified .notchpkt archive.

  • Explicit contents — no hidden chat, transcript, or project scraping.
  • Secret scanning — packets land clean or not at all.
  • Explicit lineage — stream tags and supersedes keep checkpoints ordered.
  • Client-optional — MCP agents use the store directly; people and browser chats can read the same files.
  • Your transport — Git, email, AirDrop, Tailscale, or anything else you trust.
.notch/outbox/continuation.website-refresh/ 
continuation.website-refresh/
  ├─ packet.md
  │  stream: website-refresh
  │  status: unreviewed
  │  next: verify mobile + open PR
  ├─ manifest.json   sha256 verified
  └─ artifacts/
     ├─ index.html
     └─ screenshots/

✓ Ready for the next agent.
05 · BOUNDARIES

What 3Notch isn't.

The product stays narrow on purpose. These boundaries are tested by a regression-guard in the repo.

✗No hosted relay, account, or telemetry. The store stays local; your existing transport moves packets when you choose.
✗No transcript scraping or complete-conversation claim. Fallbacks use documented hook fields and Git metadata; semantic checkpoints use context the agent explicitly supplies.
✗No automatic Git or network operations. Checkpoints stay local until you deliberately commit, push, import, or transport them.
✗No vector database or arbitrary shell through MCP. Markdown stays canonical; the server exposes selected record operations only.
